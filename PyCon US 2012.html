ABC: the mother of Python

Edit | Cancel

Before Python, Guido was a contributor to the ABC language and environment. ABC was designed to teach programming, and had many "Pythonic" ideas: integrated dicts and tuples, structure by indentation, strong typing without declarations etc. With high demand for Python in education, the legacy of ABC is more relevant than ever as we design courses and environments for learners.

Type: Talk

If Guido van Rossum is the father of Python, then ABC is the mother. ABC was a programming language and environment designed by Leo Geurts, Lambert Meertens and Steven Pemberton at CWI in the Netherlands during the 1980's. Guido was a member of the ABC team and went on to design Python to be "almost, but not quite, entirely unlike ABC".

About ABC

Steve Pemberton's definition is the best I've found:

A simple but powerful interactive programming language and environment. We did requirements and task analysis, iterative design, and user testing. You'd almost think programming languages were an interface between people and computers. Now famous because Python was strongly influenced by it.

Why ABC is relevant today

Guido departed from ABC to create a much more practical, all-purpose language. But today Python is increasingly being used also as a first programming language in schools and universities. This use as an educational tool and as a language for casual programmers can be informed by the choices made by the original ABC team, as revealed by the language implementation, it's environment, and several publications about it.

About this talk

The aim is to answer the question: What can current and future Python IDEs and libraries aimed at beginning or casual programmers learn from ABC, a language and environment explicitly designed for that audience?

The ABC language and environment will be shown live (the DOS binary can be made to work).

The main similarities and differences between Python and ABC will be discussed, often with side-by-side code comparisons.

Among the similarities, we will cover:

  • foreach style for command, with tuple unpacking;

  • block structure by indentation;

  • extensive use of dicts and tuples (called "tables" and "compounds");

  • unbounded strings;

ABC also has some features that Python did not inherit:

  • lists and dicts are only allowed to contain items of the same type;

  • variables can only be bound to a single type during one function invocation;

  • lists are automatically kept sorted (the rationale was: you either want a list to be sorted, or the ordering of the items is irrelevant);

  • numbers are either exact (represented as a ratio of two unbounded integers) or aproximate (represented as hardware floats) and division between exact numbers always produces another exact number (compare this with division in Python < 3, and the origin of one of Python's warts becomes apparent; but it is no wart in ABC, because exact numbers are rationals so 3/2 is exactly 1.5);

  • explicit files are not supported: instead, the environment auto-saves objects like functions and global variables in workspaces for use in future sessions; also, table-like global variables can be bound to standard I/O, but this binding cannot be changed within a session;

ABC also includes an editor and interactive console with auto-completion, auto-indentation and other aids. It uses the concept of workspaces which allows a form of modular programming without explicit handling of packages and imports.

The merits of these and some other features that Python dropped or changed will be discussed in the context of Python usage by people learning to program. For instance, although implicit persistency and constrained file I/O may be unacceptable to some professional programmers, there are similar features in programming systems designed for non-programmers, such as Smalltalk, Hypercard and more recently, Processing.

Audience level: Novice

Andy, thanks for your interest in reviewing my talk and posing a question.

I believe this talk is attractive from a historical point of view, and will draw an audience for that reason alone.

But I also think PyCon attendees like to learn things they can apply directly. I filed this proposal under the "Education" category and it is under this light that it should be reviewed (alas, the category does not appear in the proposal page, but only in the main review listing).

Today Python is being increasingly adopted as a teaching language. A the same time, in fundamental education, projects like Scratch gain popularity as tools to introduce programming to kids, by lowering the syntactic barrier. Python, because of its ABC inheritance, has one the lowest syntactic barriers already (LISP ans Smalltalk both have simpler syntax, but at the cost of poorer readability).

While I believe that Python is an excellent first programming language, having taught programming to beginners many times I know any language aimed at professional programmers poses some challenges to learners. For example, the fact that dicts are unordered in Python makes sense from an engineering perspective but is a poor usability decision.

One strategy to make the language more approachable is to select which features to use in examples and exercises. Another is to offer more guidance and prevent more errors through the interactive environment. A third is to make features that are useful to novice users more accessible to them. Studying ABC helps with all of these things.

My main point is that studying ABC, the ABC book and several papers written about it, where the design choices of the language are made explicit, is valuable to teachers designing Python courses aimed at beginning programmers, and to developers creating or enhancing Python tools targeting the same audience. My goal is to present a summary of what I learned from those sources to the public.
